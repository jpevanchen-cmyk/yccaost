# 野草系统 · 数据安全与隐私说明

> **文档版本**：1.6  
> **最后更新**：2026-07-13  
> **规则正本**：仓库根目录《野草系统-核心规则速查手册》**附录 J / A.1 / A.14 / A.15**  
> **读者**：自己部署野草服务器的店铺老板、运维人员

---

## 1. 这份说明是干什么的

这是给**自己开店、自己装野草**的人看的**白话版**安全说明，不是给程序员看的代码文档。

野草的想法很简单：

- **默认当坏人已经偷到了数据库**来设计；
- 目标不是「库永远偷不走」，而是**偷了也拼不回顾客隐私**；
- **数据是你的**，不是野草的，不是云的，不是联盟的。

技术细节以后由专业人员审查；这里只讲**你要懂的选择和底线**。

---

## 2. 野草怎么想安全（四句话）

1. **非必要，不透露**——能少存就少存，能少给人看就少给。  
2. **用过的信息，按规则模糊**——送完餐、结完单，详细地址等该脱敏就脱敏。  
3. **任何一处单独被偷，都拼不回完整隐私**——库、备份、云、认证中心，都不能单独「开箱即读」全部顾客信息。  
4. **保护的是顾客隐私，不是保护数据库文件本身**。

---

## 3. 部署怎么选（①②③）

野草**没有**中央大库；「云」是这家店自己的**公网门面（联络员③）**，不是平台替你存所有店的数据。

> **规则正本**：手册 **A.1 / 附录 J.1**（2026-07-13 起）。

### 三个部件（白话）

| 代号 | 名称 | 白话 |
|:---|:---|:---|
| **①** | 野草本体 | 真正的账本和程序「机房」。**不能当柜台营业**；除管理员和获授权的人外，谁都不能进 |
| **②** | 本地联络员 | 店里 WiFi/局域网的「柜台」；店内点餐、员工干活、老板日常管店都走这里 |
| **③** | 云上联络员 | 互联网上的「柜台」；外人远程点单走这里 |

| 组合 | 能不能营业 |
|:---|:---|
| 只有① | **不能**对外营业 |
| **①+②** | **标准主体**：可以本地营业（堂食、到店打包等；店内可由服务员代下外卖）。**有没有③都不影响** |
| **①+②+③** | 主体 + **可选升级**：上面全部 + 公网远程点单等 |
| ①+③（无②） | 不推荐当标准 |

**重要**：日常营业（含老板改菜、接单）**只走②或③**；进①必须是**另一条维护通道**，不能和营业入口混在一起。

**通信与出网（摘要，正本见手册 A.1.1）**

- 云和店之间：消息只走 **③↔②↔①**，③不能直接碰①；由**②主动**连③（默认每 10 秒问一次，也可改成长连接）。  
- 微信要码、查单：也由**②上网**去做，再告诉①；①更专注当账本。  
- 收款是**每一单一张新码**（网页出码用微信扫），不必做小程序。  
- **访客 WiFi 不要给上外网**，只让手机打开店里的②；上网留给店里电脑（②），免得顾客把带宽占满、②连云不稳。  
- 小店**一台路由器就够**；①②第一阶段可装同一台电脑。

### 旧说法对照

| 以前常说的 | 现在 |
|:---|:---|
| 方式 A · 纯云 | 允许但不推荐（整店①在云上） |
| 方式 B · 本地+云 | 接近「再加③」；标准主体是①+②，不是「必须有云才能开店」 |

### 断网、云挂了怎么办（摘要）

| 情况 | 野草还能干啥 |
|:---|:---|
| ③挂了或没装③，但②开着 | **本地照常**；公网客人下不了单。**不用**「切换到②」 |
| ②被关掉了 | 须在①里**手动重开②**（开店内门，不是从③切过来） |
| ①+② 且②还能上网 | 顾客连访客 WiFi（仅内网）点餐；微信付钱用顾客自己流量；②上网查单 |
| ②也不能上网 | 线上支付不可用；可现金等 |
| 纯云①且云挂了 | 只能纸单、现金等纯手工 |

## 4. 正式上线前你必须做到

对照《野草系统-正式上线前清查备忘录》，并至少包括：

| 项 | 说明 |
|:---|:---|
| **隐私加密、分开存** | 手机号、地址、姓名等隐私**加密**，和一般业务数据**分开存** |
| **HTTPS、关调试** | 对外用 HTTPS；生产环境不能开着「详细报错页」 |
| **密钥别写在代码里** | 生产密钥用环境变量等方式保管 |
| **真微信、关模拟** | 正式营业用真实微信收款；关掉演示用模拟支付 |
| **备份至少两份** | 一份在本地；一份可上传云，但必须是**加密后的**，云服务商默认读不懂内容 |

**未完成隐私加密分存，不应正式对外营业。**

## 5. 日常谁能看到什么（原则）

| 角色 | 原则 |
|:---|:---|
| 买家 | 自己的订单 |
| 卖家 | 本店经营所需；不需要顾客身份证等 |
| 骑手 | 配送所需地址/联系；不需要实名档案 |
| 联盟/认证中心 | 各管各的，**默认不看**你店订单明细 |
| 修机器的服务商 | **默认不能碰你的库**；必须你授权，且最好**限时或一次性**，用完就失效 |

---

## 6. 请服务商修机器时

- 能碰数据库、服务器的那种服务商，要有**个体户等资质**（见规则手册附录 I）。  
- **默认没有**进你数据的权限。  
- 实在要进：必须**你明确授权**——**一次性登录**或**限几小时登录**，**退出或到期就失效**。  
- 最好留记录：谁、什么时候、干了什么。

---

## 7. 备份怎么做

- **至少两份**。  
- 本地自动备份一份。  
- 可再有一份**加密后**放云盘/云存储；即使云盘被偷，也应读不出明文。  
- 可选：自己 U 盘拷一份加密备份。

---

## 8. 断网、云挂了怎么办

| 部署情况 | 线上支付 | 野草还能干啥 |
|:---|:---|:---|
| **有②且开着，③挂了或没③** | 由②上网查单；出网也断则提示失败 | **本地照常**；公网客人无法用②；访客 WiFi 仍应不上公网 |
| **②被关掉** | — | 在①**手动重开②**（开店内门，非从③切换） |
| **只有云上①** | 用不了 | 只能**纯手动营业**；提前准备的预案用上 |

## 9. 用了野草，出事谁担责？

- **你是店/server 拥有者**：怎么运营、怎么管数据、怎么合规、怎么授权服务商——**你是责任主体**。  
- **野草开发者**：提供软件和规则说明；**不对**你正式开店运营里、**不属于开发者责任范围内**的问题负责，**包括法律责任**。  
- 野草是**工具**，不是替你背锅的平台。

## 10. 常见问题

**问：我用阿里云算不算「把数据交给野草平台」？**  
答：不算。云主机是**你自己租的**，多半用来跑联络员③或试验；野草没有中央业务库收全平台数据。权威账本应在本体①（通常在店里）。

**问：必须买两台电脑吗？**  
答：**不必须**。常见是一台店内电脑跑①（及同机或同网的②）；要网络营业再租云跑③。

**问：为什么日常不能直接打开本体①或用 localhost 管店？**  
答：①是机房，不是柜台。日常营业走②或③；进①必须另开维护通道，避免机房变成第二个收银台。

**问：选纯云（①整个在云上）是不是不行？**  
答：**可以选**但不推荐；云挂了只能手工，**宜提前写好纯手动预案**。

---

**问：登录页不勾「记住用户名和密码」，浏览器还会不会自己记？**  
答：**有可能还会。** 野草未勾选时**自己不记**，并尽量不配合自动填充，但**无法 100% 禁止**浏览器弹出「是否保存密码」或自行填充。公共电脑请勿勾选野草的记住项，并建议在浏览器里关闭自动保存密码。

**问：关掉当前标签页，算不算退出？**  
答：会尽量拦截提示，并尽量通知服务器退出，**不保证每次都成功**；若没通知到，大约 **5 分钟**无「报平安」后登录会失效。超过 **15 分钟**页面无操作也会退出。要立刻退干净，请点页面上的「退出登录 / 退出工作台」。

**问：野草开源协议是什么？别人能拿去开云收费吗？**  
答：主程序是 **AGPL**（标准开源）。店铺可以**免费自建**。别人可以做收费云，但须按 AGPL 把对主程序的改动开源给使用者。另有一份**自愿认证名录**（在 GitHub 仓库 `CERTIFIED_DIRECTORY.md`）：想挂「认证」牌子才要定期承诺；不加入也能按 AGPL 用。细节见手册 **A.14**。

**问：听说以后要分成核心和插件，现在装的是那样吗？**  
答：**规则已经定好**（手册 **A.15**），但**现在程序还是整包**，还没拆成「薄核心 + 餐饮/配送插件」。不影响你先按现在方式试用；拆开是开发上的后续工作。
