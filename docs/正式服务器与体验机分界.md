# 正式服务器与体验机私人配置（定稿 · 勿再混）

> **读者**：开发者 / 智能体  
> **定稿日期**：2026-07-16  
> **目的**：完善功能时必须以「以后所有自己部署野草的人」为对象；不得把开发者私人体验机上的叫法当成通用产品规则。

---

## 1. 两套对象必须分开

| 说法 | 含义 | 是否通用产品 |
|------|------|----------------|
| **正式服务器** | 用户用野草系统**自己部署**出来的服务器 | ✅ 通用 |
| **服务器部署者** | 上述正式服务器的主人（决定是否允许他人开店、指定管理者等） | ✅ 通用 |
| **服务器管理者** | 部署者指定的账号，可进入 **「服务器设置」**（站点品牌、备案、服务器主页等） | ✅ 通用 |
| **服务器设置** | 日常整机设置入口（原误称「拥有者控制台」） | ✅ 通用 |
| **体验机 / 官方小店 / 体验日清 / 正式保留号** | 开发者**私人服务器**上为开放测试而起的配置与叫法 | ❌ 仅私人体验部署；**不是**正式服务器必有概念 |

智能体完善功能时：默认按**正式服务器 + 服务器部署者 + 服务器设置**设计。  
体验机相关逻辑必须关在「体验机开关」内，**不得**绑死到服务器设置权限上。

---

## 2. 正式服务器（通用）应有什么

- 服务器部署者（安装/运维这台机的人）  
- 是否允许他人在本机注册开店（规则可后续做开关；第一阶段可先约定）  
- 可指定一名（或多名）**服务器管理者** → 进入「服务器设置」  
- **没有**系统强制的「官方小店」「正式保留号」；部署者若自己起类似名字，那是他自己的业务叫法，不是野草通用名词  

指定管理者方式（定稿）：

- 由**超级管理员总后台**勾选「服务器管理者」，或  
- 服务器上执行：`python manage.py mark_server_owner --username=用户名`  
- **不做**网页「认领」；体验机也不要认领逻辑  

---

## 2.1 两本手册怎么分（硬规定）

| 手册 | 写什么 | 超级管理员 / 总后台 |
|------|--------|---------------------|
| **用户使用说明书**（以后《野草操作手册》） | 平时怎么点餐、管店、工作台 | **禁止出现操作步骤**；最多一句「维护见部署手册」 |
| **服务器部署手册** | 安装、更新、环境、指定管理者、救急 | **只在这里写清楚** |

原则：超级管理员只给有技术、有经验的人；普通人日常只用「服务器设置」。

---

## 3. 开发者私人体验机（仅本部署）

见同目录 `体验机与官方小店约定.md`。  

包括：体验开关、官方小店标记、体验/正式保留标记、日清、体验禁真付等。  
这些是**私人测试形态**，用于先上线收反馈；**不改手册主路线**。

---

## 4. 私人配置与 `git pull`（必须保障）

| 东西 | 在不在公开仓库 | `git pull` 会不会冲掉 |
|------|----------------|------------------------|
| 程序代码、公开文档 | 在仓库里 | 会更新成新版本（这是要的） |
| 服务器上的 `.env`（体验开关、签名钥匙、域名等） | **不进**仓库（已忽略） | **不会**被覆盖 |
| 服务器上的业务库 `db.sqlite3`（官方小店标记、管理者标记、备案号、订单等） | **不进**仓库 | **不会**被覆盖 |
| 本地 `docs/正式上线部署指南.md`、启动备忘等 | **不进**仓库 | **不会**被覆盖 |

**正确更新姿势**：云上 `git pull` → `migrate`（如有）→ 重启服务。  
不要用「整目录覆盖拷贝」替代 pull，也不要把本机库上传去覆盖云库。

体验相关**能力**写在代码里并由 `YECAO_EXPERIENCE_SITE` 等开关控制；私人**状态**（哪家是官方店、谁是管理者）存在数据库/环境变量里，与代码更新分离。

---

## 5. 对外正式发布时

- 正式服务器默认可不启用体验机能力（开关关闭即可）。  
- 以后可由服务器部署者**自行选择**是否开启类似「体验开店/日清」能力；或在对外发行版中再决定是否删除私人体验专用代码路径。  
- 无论哪种，都**不得**再把「官方小店」写成所有正式服务器的必选项。
