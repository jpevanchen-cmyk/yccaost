## 1. 项目是什么、给谁用

| 项 | 说明 |
|----|------|
| 名称 | **野草系统**（代码仓库文件夹名：YecaoST） |
| 定位 | 去中心化外卖/本地配送平台；店铺独立部署，平台不抽佣（第一阶段） |
| 第一阶段用户 | 独立店铺节点 + **店铺专属骑手** + 注册买家；**堂食为主、外卖为辅** |
| 终端 | 响应式网页（手机 + 电脑同一套） |

---

## 2. 技术取向

| 项 | 选择 | 理由 |
|----|------|------|
| 语言 | Python 3.11 | 开发者环境已具备 |
| 框架 | Django 5.2 | 项目已搭建，含用户、订单、菜品等基础模块 |
| 数据库 | SQLite（现阶段） | 开发简单；单店冷启动够用；后续可按方案迁移 |
| 环境 | 项目内 `.venv` 虚拟环境 | 与系统 Python 隔离 |
| 支付 | 微信 Native 扫码（已实现接入骨架）+ 模拟支付 + 现金 | 各店自有商户号；回调或轮询更新订单状态 |
| 打包分发 | **不适用 Inno 桌面安装包** | 网页项目，按服务器部署说明交付 |

## 5.4 店铺工作台与登录双门（A.2.1 实现摘要）

| 项 | 说明 |
|----|------|
| 生态门 | `/accounts/login/` — 买家、店主；子账号登录视为无效账号 |
| 店铺工作台 | `/s/<shop_code>/work/` — 店主 + 本店员工；三 Tab（服务员/后厨/骑手） |
| 登录状态 | **两套独立会话**：生态用 Django 登录；工作台用独立会话字段；退出入口分开 |
| 店铺专属骑手 | 工牌子账号；本店派单；店铺工作台骑手 Tab |
| 野草自由骑手 | 生态个人账号；第二阶段区域抢单池（手册 A.2.2、B.3） |
| 管店后台 | `/seller-panel/` — 店主生态登录进入；**不含**店内总台 |
| 员工管理 | `/seller-panel/workbench/` — 二维码、子账号、外卖交接方式 |
| 店主在工作台 | 三 Tab 均可操作 |
| 员工在工作台 | 三 Tab 可看概况；仅本岗位可操作 |
| 旧地址 | `/owner-hub/`、`/waiter-home/` 等自动跳转本店工作台（有店码时） |
| 关页守护 | 尽量拦截离开提示 + 尽量发退出；约 5 分钟无报平安失效；15 分钟无操作退出 |
| 记住密码 | 两登录页可勾选记住用户名+密码（默认不勾）；**无法 100% 禁止浏览器自己记密码** |

## 5.5 分层编号（A.13 摘要 · 智能体必读）

> **正本**：`野草系统-核心规则速查手册.md` · **A.13**；附录 B.1.2、C.0.1 为衔接摘要。

| 项 | 规则 |
|----|------|
| 设计前提 | **唯一认证中心**为理想形态；多认证中心后并入同一体系时，合并撞车**共用 `-A01` 顺延** |
| 第一阶段 | 开店**自动生成店铺码**（本机唯一）；**无**服务器码 / 联盟码 |
| 第二/三阶段 | 向联盟 / 认证中心报到时**补字段**，不重生店铺码 |
| 改码权限 | **仅合并登记**可改短码；卖家与普通人**不可手改** |
| 订单展示 | 现查 `ShopProfile.shop_code`；**合并后站内历史订单统一显示新店铺码** |
| **站外例外** | **已发微信等第三方的付款说明不会随改码自动变**（智能体须牢记） |
| 短号 | UUID 前 8 位，**不保证全球唯一**；完整 `order_id` 为认账依据 |
| 合并去重 | `原码-A01`、`原码-A02`… 再合并则**顺延**；联盟码 / 服务器码 / 店铺码**同一算法** |
| 代码 | 须抽**公共函数**实现合并重命名；店铺自动生成**已实现**（`shop_code_helpers.py`） |
