## 1. 项目是什么、给谁用

| 项 | 说明 |
|----|------|
| 名称 | **野草系统**（代码仓库文件夹名：YecaoST） |
| 定位 | 去中心化外卖/本地配送平台；店铺独立部署，平台不抽佣（第一阶段） |
| 第一阶段用户 | 独立店铺节点 + **店铺专属骑手** + 注册买家；**堂食为主、外卖为辅** |
| 终端 | 响应式网页（手机 + 电脑同一套） |

---

## 2. 技术取向

| 项 | 选择 | 理由 |
|----|------|------|
| 语言 | Python 3.11 | 开发者环境已具备 |
| 框架 | Django 5.2 | 项目已搭建，含用户、订单、菜品等基础模块 |
| 数据库 | SQLite（现阶段） | 开发简单；单店冷启动够用；后续可换库。**表结构起点为「数据架构 v1」**（见下 §3） |
| 环境 | 项目内 `.venv` 虚拟环境 | 与系统 Python 隔离 |
| 支付 | 微信 Native 扫码（已实现接入骨架）+ 模拟支付 + 现金 | 各店自有商户号；回调或轮询更新订单状态 |
| 打包分发 | **不适用 Inno 桌面安装包** | 网页项目，按服务器部署说明交付 |

## 3. 当前代码概况（以 2026-07-14 数据架构 v1 为准）

> **数据架构 v1（2026-07-14）**：已将历史上多份逐步迁移（旧 `0001`～约 `0025`）**压平为一条初始迁移** `waimai/migrations/0001_initial_v1.py`。  
> **含义**：以当时已落地的全部表结构为**新仓库的数据库起点**；旧迁移链不再保留，不作为升级路径。新环境一律空库执行该初始迁移即可。  
> **扫描范围**：`waimai`、`wuwei_system`、`waimai/payments/`、卖家/买家主要模板等。细项见本方案后文章节与拥有者侧私有进度文档（公开仓不收录）。

### 已有（主体已落地）

- Django 项目骨架：`wuwei_system`（配置）、`waimai`（业务应用）；`.venv`、`requirements.txt`
- 买家注册 / 登录 / 登出（注册仅买家）；买家「我的」分当前/历史订单；全角色可修改本人密码
- 店铺「注册服务器」（`ShopProfile` 等）
- 点菜页 `/shop/`、无刷新购物车加减删、下单、订单历史、订单详情
- **遗留**：独立公共名录页 `/directory/`（**与现行规则不符**，待改为展示主页 + 名录块，见 A.10）
- 取餐方式：外卖 / 堂食 / 打包（到店单不收配送费、不派骑手；含到店付现金、接单、未收款结案等）
- 卖家面板：菜品、订单、骑手、配送费、支付等分区
- 店铺可配置配送费与减免（`ShopDeliverySettings` + `delivery_helpers`）
- 备货流程；本店**自动派单**（手头未完成单最少优先）+ **手动指派/改派**（`dispatch_helpers`）
- 店铺员工子账号统一分为**管理职务 / 普通员工**；店主自填职务名、勾选权限；饮食插件可提供服务员/后厨/骑手预置模板
- **店铺工作台** `/s/店铺码/work/`：扫码登录；主体始终有「订单处理」（仅看单/改状态/确认收款/联系顾客）；业态 Tab 由已启用插件加载；订单可切新单/旧单优先
- **员工工作台管理**（`/seller-panel/workbench/`）：二维码、子账号、职务与权限、考勤、协作设置
- 野草生态登录 `/accounts/login/` **仅**买家、店主；子账号不可从此门登录
- 同账号只允许一处在线：新登录后旧设备退出并提示；不同账号不互踢
- 支付：模拟支付、微信 Native **APIv2**（下单/查单/验签/异步通知）、现金确认；`ShopPaymentSettings`、`PaymentRecord`。**JSAPI、刷脸、通用结算/分账尚未实现**；新商户当前官方接入以 APIv3 为主，APIv2 适配须核实与真付验证
- 手机端样式与脚本（响应式基础）
- **A.11 第一批已落地**：营业时段 / 渠道开关 / 打烊暂停；桌台桌码与虚拟码池；桌码主单加点合并；菜单清单（新建、复制、切换使用、可售上限、按清单控制会员价/特价展示）
- 商品管理增强：会员价 / 特价按百分比或固定价；新商品默认同步进各清单但可选“全部上架”；特价每人限购默认 1；菜品销量日 / 周 / 月排行
- 买家点菜页已改为**同一商品按通用价 / 会员价 / 特价分条展示**，每条独立“加入购物车”按钮，并自动显示购买条件说明
- 卖家后台全站未保存修改拦截（切页 / 切清单 / 返回时提示保存、放弃或取消）
- 订单管理已支持**关键词搜索 + 状态筛选 + 近 1 个月默认时间范围**
- 订单对外展示号已改为 **`短号-店铺码-YYYYMMDD`**（现阶段店铺码为空时显示 `短号-YYYYMMDD`）；`ShopProfile.shop_code` 已预留；**编号分层与合并规则**见手册 **A.13**（规则已定，自动生成与合并公共函数待代码）
- 服务器拥有者目前可在超级管理员后台维护整台服务器的 ICP / 公安备案号，并上传公安备案审核平台为本网站提供的图标；系统自动生成官方查询链接，在全站页脚显示「图标 + 公安备案号」；普通店主无权修改，不依赖先开店
- 数据模型含部分第二阶段占位字段，**业务未启用**

## 5.5 分层编号（A.13 摘要 · 智能体必读）

> **正本**：`野草系统-核心规则速查手册.md` · **A.13**；附录 B.1.2、C.0.1 为衔接摘要。

| 项 | 规则 |
|----|------|
| 设计前提 | **唯一认证中心**为理想形态；多认证中心后并入同一体系时，合并撞车**共用 `-A01` 顺延** |
| 第一阶段 | 开店**自动生成店铺码**（本机唯一）；**无**服务器码 / 联盟码 |
| 第二/三阶段 | 向联盟 / 认证中心报到时**补字段**，不重生店铺码 |
| 改码权限 | **仅合并登记**可改短码；卖家与普通人**不可手改** |
| 订单展示 | 现查 `ShopProfile.shop_code`；**合并后站内历史订单统一显示新店铺码** |
| **站外例外** | **已发微信等第三方的付款说明不会随改码自动变**（智能体须牢记） |
| 短号 | UUID 前 8 位，**不保证全球唯一**；完整 `order_id` 为认账依据 |
| 合并去重 | `原码-A01`、`原码-A02`… 再合并则**顺延**；联盟码 / 服务器码 / 店铺码**同一算法** |
| 代码 | 须抽**公共函数**实现合并重命名；店铺自动生成**已实现**（`shop_code_helpers.py`） |
